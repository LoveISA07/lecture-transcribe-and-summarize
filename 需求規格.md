# 課程錄音自動化整理工具 - 需求規格

> 版本：v1.0｜日期：2026-06-27

---

## 一、專案目標

將上課錄音（MP3 / M4A / WAV）自動轉成兩份 Markdown 結構式筆記：

- **逐字稿**（可閱讀、有章節時間軸）
- **重點整理**（含邏輯地圖、章節摘要、詞彙表、錯字對照）

支援同一堂課被分成多段錄音的情況，自動合併處理。

---

## 二、硬性約束

1. **不使用任何付費 API key**（不用 Groq、OpenAI、DeepSeek 等）
2. **不使用 Colab 或雲端 GPU 服務**
3. **單一工具一次跑完**，不要分兩步驟（不要「先在 A 工具轉好，貼到 B 工具處理」這種流程）
4. **可在文書筆電（舊 CPU）運行**

---

## 三、輸入

支援兩種輸入模式：

### 模式 A：單一檔案
```
路徑：D:\上課錄音\2026-06-27, 糖尿病概論.mp3
```

### 模式 B：資料夾（含多段錄音）
```
路徑：D:\上課錄音\2026-06-27\
內容：
  ├── 2026-06-27, 跑者營養1.mp3
  ├── 2026-06-27, 跑者營養2.mp3
  └── 2026-06-27, 跑者營養3.mp3
```

---

## 四、輸出

每次處理產出 2 個 Markdown 檔，**放在輸入檔案（或資料夾）的同一層**：

- **檔案 1**：`<主題>-逐字稿.md`
- **檔案 2**：`<主題>-重點整理.md`

**主題命名規則**：從檔名提取，去除日期前綴、段別後綴
- 範例：`2026-06-27, 跑者營養1.mp3` → 主題 = `跑者營養`
- 範例：`糖尿病_part1.mp3` → 主題 = `糖尿病`

---

## 五、功能需求

### 5.1 音訊預處理

- 統一轉成 16kHz 單聲道 WAV
- 長檔（> 25MB）自動切段（10 分鐘/段，5 秒 overlap）
- 使用 ffmpeg

### 5.2 語音轉文字（ASR）

- 使用 **本地 faster-whisper** 或 **whisper.cpp**
- 預設模型大小：`base`（文書筆電可跑）
- 可選模型：`tiny`（更快但品質差）、`small`（更慢但品質好）
- 語言：繁體中文（`zh`）
- 輸出格式：SRT（含時間軸）
- 自動合併多段 SRT，時間軸連續

### 5.3 檔案分群（資料夾模式才需要）

#### 支援的檔名模式

| 模式 | 範例 | 解析結果 |
|------|------|----------|
| 數字後綴 | `跑者營養1.mp3`、`跑者營養2.mp3` | topic=跑者營養, part=1/2 |
| 中文數字 | `跑者營養一.mp3`、`跑者營養二.mp3` | part=1/2 |
| Part 標記 | `跑者營養 part 1.mp3` | part=1 |
| 括號標記 | `跑者營養(1).mp3` | part=1 |
| 字母後綴 | `跑者營養A.mp3` | part=A |
| 日期前綴 | `2026-06-27, 跑者營養1.mp3` | date=2026-06-27, topic=跑者營養 |

#### 分群規則

- 相同日期 + 相同主題 → 視為同一堂課
- 自動排序（依段別編號 1, 2, 3... 或 A, B, C...）

#### 使用者確認（必要）

- 工具偵測到群組後，**必須在終端機顯示群組資訊並詢問 y/n**
- 詢問內容包含：群組名稱、檔案清單、總時長、預計輸出
- `Y` = 合併處理
- `n` = 跳過
- `split` = 當作獨立檔案分別處理

### 5.4 多檔 SRT 合併

合併多段 SRT：

- 時間軸**連續計算**（Part 2 從 Part 1 結束時間接續）
- 每段字幕開頭加 `[Part X]` 前綴
- 範例：

```srt
1
00:00:00 --> 00:00:15
[Part 1] 各位同學大家好...

2
00:45:23 --> 00:45:40
[Part 2] 好，我們繼續剛剛的話題
```

### 5.5 LLM 結構化（4 個 Pass）

#### Pass 0：領域偵測

- 輸入：SRT 前 1000 字
- 輸出：領域 JSON（包含 `sub_domain`、`expected_vocabulary`、`common_asr_errors`）

#### Pass 1：章節骨架

- 輸入：完整 SRT + 領域 context
- 輸出：章節 JSON（含標題、時間範圍、關鍵主題）

#### Pass 2A：產出逐字稿

- 輸入：SRT + 章節骨架 + 領域 context
- 輸出：「逐字稿」Markdown

#### Pass 2B：產出重點整理

- 輸入：SRT + 章節骨架 + 領域 context
- 輸出：「重點整理」Markdown

## 六、輸出格式詳細規格

### 6.1 檔案 1：`<主題>-逐字稿.md`

```markdown
# 課程逐字稿：<自動推測主題>

> 來源：xxx.mp3｜時長：HH:MM:SS｜整理：YYYY-MM-DD
> 來源檔案：[列出所有輸入檔名]

## <章節標題> [Part X｜HH:MM]

<段落 1，約 300-400 中文字>

<段落 2，約 300-400 中文字>

## <下一章節標題> [Part X｜HH:MM]
...
```

**規則：**

- 每段 300-400 中文字（理想值，不超過 500）
- 標題層級用 `##`
- **時間戳只在章節標題**，標題以下不標時間
- 標題格式：`## 標題 [Part X｜HH:MM]`（單檔時省略 Part）
- 加全形標點符號：，。、！？：；（）""''
- 去除口頭禪（那個、就是、嗯、對、然後）
- 核心專有名詞、藥物名、數據用 **加粗**
- 禁止條列、表格、邏輯地圖
- 禁止摘要或重點整理
- 保留原始內容，不刪減

### 6.2 檔案 2：`<主題>-重點整理.md`

```markdown
# 課程筆記：<自動推測主題>

> 來源：xxx.mp3｜時長：HH:MM:SS｜整理：YYYY-MM-DD
> 來源檔案：[列出所有輸入檔名]

## 一、課程摘要

<80-150 字的總覽>

## 二、課程邏輯地圖

​```mermaid
graph TD
    A[主題] --> B[子題1]
    A --> C[子題2]
    ...
​```

## 三、章節重點

### 第 1 章 <標題> (Part X｜HH:MM - HH:MM)

**核心論點**：<一句話>

**重點條列**：

- ...

**詳細說明**：<2-4 句連貫>

### 第 2 章 ...

...

## 四、關鍵詞彙表

| 詞彙 | 白話解釋 |
|------|----------|
| ... | ... |

## 五、辨識錯字對照

| 原文 | 推測正確 | 出現章節 |
|------|----------|----------|
| 醫療素 | 胰島素 | 第 2 章 |
```

**規則：**

- 必須有 5 大區塊：摘要、邏輯地圖、章節重點、詞彙表、錯字對照
- Mermaid 圖節點用 2-6 字簡短詞，不超過 12 個節點
- 不可編造內容，不確定處加註 `⚠️ [需查證]`
- 核心專有名詞、藥物名、數據用 **加粗**
- 繁體中文，台灣用詞
- 醫學術語首次出現時用全形括號加註原文

## 七、技術要求

1. **語言**：Python 3.8+
2. **必要套件**：
   - `faster-whisper` 或 `openai-whisper`
   - `ffmpeg`（subprocess 呼叫）
3. **可選套件**：Mermaid 渲染（如需預覽）
4. **LLM 整合**：由實作者決定如何呼叫 LLM（本地 Ollama、外部 API、Antigravity 自身能力等）
5. **領域詞典**：JSON 檔，可擴充
6. **CLI 介面**：

```powershell
# 完整流程
python lecture_to_notes.py "input.mp3" [--domain 糖尿病]

# 處理資料夾（自動分群 + 詢問確認）
python lecture_to_notes.py "D:\folder\"

# 跳過 ASR（已先手動跑過）
python lecture_to_notes.py "input.srt" --skip-asr

# 只跑 ASR
python lecture_to_notes.py "input.mp3" --asr-only

# 指定領域詞典
python lecture_to_notes.py "input.mp3" --domain 糖尿病
```

## 八、領域詞典（雙軌設計）

### 軌道 1：內建詞典（可選）

JSON 格式，可預載特定領域：

```json
{
  "糖尿病": {
    "expected_terms": ["HbA1c", "SGLT2i", "GLP-1", "Metformin", "胰島素阻抗"],
    "asr_corrections": {
      "醫療素": "胰島素",
      "糖化血色速": "糖化血色素"
    },
    "logic_map_hints": "治療目標 → 藥物分類 → 機轉 → 副作用 → 追蹤"
  }
}
```

### 軌道 2：自動偵測（必備，Pass 0 執行）

- 沒對應詞典時，用 LLM 自動生成的 `expected_vocabulary`
- 確保新領域也能運作

## 九、邊界情境

| 情境 | 處理方式 |
|------|----------|
| 輸入是資料夾但只有 1 個檔案 | 視為單檔處理，不顯示群組提示 |
| 群組只有 1 個檔案 | 視為單檔，不加 `[Part X]` 標記 |
| 使用者拒絕合併（選 `n`） | 改為獨立處理每個檔案 |
| ASR 跑很久（> 30 分鐘） | 顯示進度條 |
| SRT 內容為空 | 警告，繼續處理空結構 |
| 不同主題混在同一資料夾 | 顯示所有群組，讓使用者選 all / select / skip |
| LLM API 呼叫失敗 | 顯示錯誤訊息，提供手動 fallback 選項 |
| Mermaid 語法錯誤 | LLM 重試一次，失敗時改用純文字描述邏輯結構 |

## 十、不在範圍內

以下功能**不需要實作**：

- 視訊畫面 OCR（純音訊，無視覺）
- 講者分離（不區分多人）
- 自動翻譯
- 雲端同步
- 即時轉錄（離線批次處理）

## 十一、驗收標準

工具完成後，以下測試案例必須通過：

1. **單檔測試**：1 小時中文錄音 → 產出 2 個 .md，內容連貫、無重大錯漏
2. **多檔測試**：`X1.mp3` + `X2.mp3` → 自動合併，產出 1 組檔案，時間軸連續
3. **分群確認測試**：資料夾有 2 個群組 → 終端機顯示群組清單，詢問 y/n
4. **領域偵測測試**：醫學錄音 → 自動偵測為「醫學」並套用對應邏輯
5. **錯字校正測試**：含 ASR 錯字（「醫療素」）→ 在「錯字對照」表格中標示
6. **時間軸測試**：合併檔案的 `[Part 2]` 時間戳從 Part 1 結束處接續
7. **段落長度測試**：逐字稿每段 300-400 中文字
8. **時間戳位置測試**：時間戳只在章節標題，內文不出現時間戳

## 十二、實作自由度

由於「不使用 API key、不使用 Colab」的限制，Antigravity 需自行評估並決定：

- **選項 A**：完全本地（Whisper + Ollama 跑 LLM）— 慢但 0 成本
- **選項 B**：本地 Whisper + 呼叫 Antigravity 自身 LLM 能力（如果支援）
- **選項 C**：混合方案（具體由 Antigravity 評估）

**LLM 提示詞模板（給 Antigravity 參考）：**

所有 LLM 呼叫都用以下 4 個 prompt 模板（細節見第五節各 Pass 說明）：

- `prompts/pass0_domain_detection.txt`
- `prompts/pass1_chapter_skeleton.txt`
- `prompts/pass2a_transcript.txt`
- `prompts/pass2b_notes.txt`

具體 prompt 內容由實作者依本規格設計，須符合第六節的輸出格式要求。
